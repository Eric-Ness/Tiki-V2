# v0.6.5

Blank-screen hotfix. Unblocks every fresh-install and post-auto-update v0.6.4 user sitting in a blank window. Bundles two occurrences of the same React-19 fresh-reference bug class, plus a structural guard so a single component crash can no longer take down the whole app.

## Fixes

- **Blank screen — unstable Zustand selector fallbacks** ([#210](https://github.com/Eric-Ness/Tiki-V2/issues/210)) — Nine selectors across 8 components used inline `?? []` / `?? {}` fallbacks. When the project key was missing from the store (the fresh-install / no-active-project state), each call returned a freshly-allocated reference, so React 19's `useSyncExternalStore` saw a "new" snapshot every check and re-rendered until tripping the max-update guard. `<KanbanBoard>` mounted first and threw; with no error boundary above it, React unmounted the entire tree → blank WebView in production (no DevTools surface to show the error). Lifted `EMPTY_TABS` and `EMPTY_COLUMN_ORDER` to module-level constants exported from `terminalStore.ts` and `kanbanStore.ts`; all 9 callsites updated to use them.

- **Blank screen v2 — unstable hook arguments** ([#212](https://github.com/Eric-Ness/Tiki-V2/issues/212)) — The same bug class, one indirection further out: fresh `{}` / `[]` passed as *hook arguments* in `App.tsx` (not inside a selector) destabilized `useEffect` dependencies and looped `<KanbanBoard>` again. Lifted `EMPTY_ACTIVE_WORK` and `EMPTY_RECENT_ISSUES` to module scope and replaced all 4 fresh-allocation sites (`1× activeWork ?? {}`, `3× recentIssues || [] / ?? []` — including one in the project-specific-cwd branch of `loadState` that wasn't in the original issue report and was caught by a verification grep).

- **Crash containment — nested ErrorBoundary** ([#212](https://github.com/Eric-Ness/Tiki-V2/issues/212)) — New `ErrorBoundary` class component (`getDerivedStateFromError` + `componentDidCatch`, optional `fallback` render-prop and `label`, default fallback shows error name/message + a Reload button that resets boundary state). The panel group is now wrapped with nested boundaries: an outer `center-pane` boundary plus per-view inner boundaries (`terminal` / `kanban` / `dependencies` / `settings`). A crash in one panel now falls back to a recoverable UI instead of unmounting the whole tree — the exact failure mode that turned #210 and #212 into blank screens.

## Notes

- The blank-screen bug class was latent since v0.5.x; masked for users with a non-null persisted `activeProjectId`. The v0.6.3 → v0.6.4 auto-update reset that state for many users, surfacing it as a v0.6.4 regression.
- Tests: shared 65 / desktop 122 / framework 10 — all green, unchanged from the v0.6.4 baseline (behavioral fixes, no new test surface). `pnpm build` + typecheck clean; `cargo check --tests` clean.
- Also landed in this tag, but separate from the blank-screen bug class: a framework fix ([#211](https://github.com/Eric-Ness/Tiki-V2/issues/211)) so the Kanban pipeline reflects `/tiki` workflow state transitions (worktree-aware `state.mjs` path resolution + per-step shim invocations in `yolo.md` + a regression test). Not part of the blank-screen hotfix story, but shipped on the same tag.
