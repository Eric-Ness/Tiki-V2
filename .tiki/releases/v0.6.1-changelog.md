# v0.6.1

**UX polish continued** — six interface-design improvements scoped during a 2026-05-15 UI deep-dive, plus the #180 Claude-CLI resolver bug fix that shipped to main earlier the same day.

## Features

- **Status-aware Resume button on Active Work cards** ([#186](https://github.com/Eric-Ness/Tiki-V2/issues/186)) — Sidebar `WorkProgressCard` now has a primary action button that switches to the Terminal view, activates the associated terminal tab (or creates one), and writes the next-step tiki command (`/tiki:get`, `/tiki:plan`, `/tiki:execute`, `/tiki:ship`, etc.) — derived from the work's current pipeline state. Pairs with the E10 terminal-association tracking from v0.6.0.
- **Shared `<Skeleton>` primitive + 3 adoption sites** ([#183](https://github.com/Eric-Ness/Tiki-V2/issues/183)) — New `components/ui/Skeleton.tsx` with `<Skeleton>`, `<Skeleton.Text lines={N}>`, and `<Skeleton.Card>` composites. Adopted in the sidebar issues list (initial fetch), the IssueDetail plan area (newly-instrumented `loadingPlan` flag — previously rendered nothing during the IPC), and IssueComments. Shimmer gradient is gated inside `@media (prefers-reduced-motion: no-preference)` so it's never painted under reduced-motion.
- **Toast action buttons** ([#185](https://github.com/Eric-Ness/Tiki-V2/issues/185)) — `Toast` API gains an optional `action?: { label, onClick }` rendered as a compact button before the dismiss control (try/catch + finally always dismisses). First consumer: the framework-outdated warning toast now carries an inline "Update" button that triggers the existing install flow via a `tiki:install-framework` custom DOM event.
- **Real-time inline validation across `IssueFormModal` fields** ([#184](https://github.com/Eric-Ness/Tiki-V2/issues/184)) — Extends the existing title-required pattern to the custom-branch input (required + practical git check-ref-format validation: blocks leading `-` / `.`, trailing `.lock`, `..`, whitespace, and `~ ^ : ? * [ \ @`) and adds a hint beneath the description when it's empty and the Enhance-with-AI button is disabled.
- **Step-specific icons in Pipeline Timeline** ([#181](https://github.com/Eric-Ness/Tiki-V2/issues/181)) — Replaces the bare 1–6 digits in `PipelineTimeline` step circles with inline Heroicons-style SVGs: cloud-arrow-down (GET), magnifying-glass (REVIEW), bullet-list (PLAN), clipboard-check (AUDIT), play triangle (EXECUTE), paper-airplane (SHIP). Completed state keeps the ✓ checkmark pop. Also closes a pre-existing tooltip gap — circles now carry `title={step}` for hover discoverability.
- **Illustrated empty state for Kanban "No issues" columns** ([#182](https://github.com/Eric-Ness/Tiki-V2/issues/182)) — Replaces the bare `<div>No issues</div>` with a soft tray-glyph SVG + label at 50% opacity, mirroring v0.6.0's `DetailEmptyState` pattern. Suppressed during drop-over so the amber drop-target ring remains the sole drop signal.

## Bug Fixes

- **Enhance-with-AI button mislabel: "CLI not installed" when Claude Code was on PATH** ([#180](https://github.com/Eric-Ness/Tiki-V2/issues/180)) — Windows GUI processes inherit `explorer.exe`'s *login-time* PATH, so a `claude` installed (or PATH-registered) after the last login was invisible to the Tauri process even though a fresh terminal saw it. New `resolve_claude_cli()` in `github.rs` tries PATH first, then probes known per-user install locations (`%APPDATA%\npm\claude.cmd`, `%USERPROFILE%\.local\bin\claude.exe`, `~/.claude/local`, `%LOCALAPPDATA%\Programs\claude`, plus Unix equivalents). Shared by `check_claude_cli` and `enhance_issue_description` so the button-enabled state always matches what actually runs. UI label softened "CLI not installed" → "CLI not found".
