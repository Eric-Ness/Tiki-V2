# v0.5.7 — Terminal clipboard repair

Single-issue follow-up to v0.5.6. Restores terminal copy/paste, which had been fully non-functional since the v0.5.3 terminal-polish bundle.

## Bug Fixes

- **Ctrl+V paste and Ctrl+C copy restored in the terminal** ([#171](https://github.com/Eric-Ness/Tiki-V2/pull/172)). As of v0.5.6 neither shortcut did anything useful in a focused terminal:
  - **Ctrl+V** had no handler at all. #155 (v0.5.3) deleted the custom paste handler on the premise that xterm.js's native `paste`-event listener would own paste — but that listener does not fire reliably inside the Tauri WebView2, leaving paste dead with no fallback. The fix adds an explicit `Ctrl+V` / `Ctrl+Shift+V` handler that reads the clipboard via `navigator.clipboard.readText()` and hands it to `xterm.paste()`, which still honors the shell's bracketed-paste mode.
  - **Ctrl+C** never had a copy binding — only `Ctrl+Shift+C` copied, and plain `Ctrl+C` always sent SIGINT even with an active selection. The fix adopts the Windows Terminal / VS Code convention: `Ctrl+C` copies when a selection exists, and falls through to SIGINT when it doesn't.
  - **Double-paste guard.** A capture-phase `paste` listener on the terminal's parent container runs before the event reaches xterm's helper textarea, so `stopPropagation()` blocks xterm's native listener from also writing. This makes the custom `Ctrl+V` handler the single paste path and structurally prevents the #155 double-paste regression — regardless of whether a given WebView2 dispatches the native event.

  The fix is confined to `apps/desktop/src/components/terminal/Terminal.tsx` and adds no new dependencies.

## Known limitations

- **`tauri-plugin-clipboard-manager` was deliberately deferred.** The issue body recommended adding the plugin so clipboard reads/writes go through the OS clipboard via Rust, bypassing WebView2 permission gating. That requires an elevated `pnpm install` to regenerate `pnpm-lock.yaml` — the same Windows reparse-point blocker that deferred #161 out of v0.5.4 — so this release uses the no-dependency `navigator.clipboard` path instead. The fix was smoke-tested on the dev machine and confirmed working before tagging; if `navigator.clipboard.readText()` is found to be blocked in some other WebView2 environment, the plugin path is the documented next step.

## Release process notes

- Unlike v0.5.6, this release was **smoke-tested before the tag was pushed** — Ctrl+V single-paste, Ctrl+C copy/SIGINT split, and the Ctrl+Shift+C regression check were all manually verified against the `fix/171-terminal-clipboard` branch first.
