# v0.5.2

Theme: **dependency graph richness — Bundle A**. Four visual upgrades that turn the Dependency Graph from a static topology diagram into something that telegraphs work state at a glance: variable node sizes by plan weight, edge animation that means "work is flowing here right now", GitHub label color encoding, and a hard/soft edge distinction.

All four issues touch the same module (`apps/desktop/src/components/dependencies/`) but different concerns — they were sequenced #152 → #151 → #153 → #154 to minimize file-region overlap. No Rust changes were strictly necessary (one was investigated and turned out unneeded — see #153 below).

## Added

- **Node height scales with plan phase count** (#152) — Dep graph nodes used to be exactly 200×60px regardless of whether the issue was a one-line typo fix or a 12-phase epic. Plan-file phase count is now folded into node visual height via a clamped staircase: 1 phase → 50px, 2-3 → 60px (the default), 4-6 → 75px, 7+ → 90px. `useDependencyGraph.ts` fetches plans in parallel with issue bodies via the existing `get_plan` Tauri IPC. Dagre's `setNode` gets matching heights and the y-centering math switched from the hardcoded `pos.y - 30` to `pos.y - h/2` so swimlane rank-grouping from #149 stays intact. Width stays at 200 (variable widths would have broken the rank-grouping; #149 rounds y, not x). Issues with no plan yet render at the default 60px.

- **Edges animate when work is actively flowing through them** (#151) — ReactFlow's animated-dash edge style was set to `animated: false` everywhere except Critical Path mode, wasting a strong visual signal. Now `animated: true` whenever an edge's source is `completed`/`closed` AND its target is `executing` — i.e. "work is in flight along this edge right now." A status lookup map built from the already-resolved node data feeds the per-edge decision, so no new fetches. Critical Path and hover-lineage modes still override (they explicitly set `animated`) — those are deliberate view-mode focuses that intentionally suppress flow noise. All-completed and all-pending releases stay static, correctly.

- **GitHub labels render as colored chips below each issue title** (#153) — The Rust `fetch_github_issue_by_number` was already returning `labels: Vec<GitHubLabel>` with name + color; the frontend just wasn't consuming them. Now `FetchedIssue` and `IssueNodeData` carry the labels through, and `IssueNode` renders a row of 10×6px colored pills using each label's GitHub-canonical hex color. Hovering a chip reveals the name via the native `title` attribute. Caps at 4 visible chips with a `+N` indicator (whose title aggregates the overflow names) — past four the row gets unreadable as encoding. Issues with no labels render no chip row and no empty space.

- **Hard dependencies render solid, soft references render dashed** (#154) — Direct follow-up to #150's parser extension. `parseDependencies` previously returned `number[]` and lost the semantic distinction: a hard "depends on #N" (blocks merge) and a soft "## Related — #N" (same code area, not blocking) rendered as identical solid arrows. Return type is now `{ number, kind: 'hard' | 'soft' }[]`. Hard triggers: `depends on`, `blocked by`, `requires`, `after`. Soft triggers: `related to`, `see also`, and all `## Related` section scans. If the same target appears as both hard and soft, hard wins — the stronger claim takes precedence. `useDependencyGraph` sets `strokeDasharray: '6 4'` on soft edges; `DependencyGraph.tsx`'s style overlays now spread `...edge.style` so the dashed pattern survives Critical Path / hover-lineage modes (previously they would have replaced the entire style object and dropped the dasharray). Legend gained solid + dashed swatches.

## Notes

- The Windows pnpm reparse-point block (CLAUDE.md) was still in effect, so `pnpm build` could not run end-to-end this session. Rust changes verified via `cargo check` (1-3s incremental, clean). TypeScript verified by inspection plus a node REPL pass on the parser's edge-kind logic — 7 cases including hard-wins-over-soft, soft-only section scans, cross-release filtering, and mixed bodies. Visual verification of the desktop app still pending (the next session should launch `pnpm tauri:dev` from the x64 Native Tools Command Prompt and eyeball v0.5.1's swimlanes/hover-lineage AND v0.5.2's new node sizing/chips/dashed edges).
- #153 had a Rust-side change planned that turned out to be unnecessary — the data was already on the wire, just untyped at the TS boundary. A common "phantom dependency" shape in cross-language stacks worth being skeptical of when scoping work.
- #154's `...edge.style` spread in the lineage and critical-path overlays is technically a latent bug fix beyond #154 itself: if any future feature adds a base edge style (a per-edge marker color, a self-edge border), the overlays would have silently dropped it under those view modes. The spread makes the overlays composable.
