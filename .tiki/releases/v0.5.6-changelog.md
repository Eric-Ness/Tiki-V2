# v0.5.6 — Clickable terminal URLs restored

Single-issue follow-up to v0.5.5 fixing the most visible symptom from the v0.5.5 smoke test: terminal URL spans were no longer clickable after the `xterm` → `@xterm/xterm@5.5.0` namespace migration.

## Bug Fixes

- **Clickable URLs in terminal output restored** ([#169](https://github.com/Eric-Ness/Tiki-V2/pull/170)). Added `allowProposedApi: true` to the `XTerm` constructor in `apps/desktop/src/components/terminal/Terminal.tsx`. The legacy `xterm@5.3.0` exposed several APIs unconditionally that `@xterm/xterm` 5.4+ gates behind this flag — `@xterm/addon-web-links@0.12.0` uses one of them internally to register URL click handlers. Without the flag, `loadAddon` succeeded but the addon's proposed-API calls returned early, so URL spans never got a click handler attached. This regression was introduced by #161 (shipped in v0.5.5) and the v0.5.5 changelog's "no behavior change expected" claim was therefore incorrect.

## Known limitations

- **Ctrl+V and Ctrl+Shift+C status remains unverified.** The v0.5.5 smoke test also flagged clipboard shortcuts as potentially not working, but without an A/B test against v0.5.4 it's not yet clear whether these are regressions from the namespace swap or pre-existing gaps. The fix in this release intentionally does **not** touch keyboard handling — changing it risks re-introducing the #155 double-paste regression. If clipboard shortcuts are still broken after v0.5.6 lands, a follow-up issue will scope the investigation properly.

## Release process notes

- This was a *speculative* fix: shipped without the v0.5.4 A/B test that would have confirmed the regression hypothesis directly. The `allowProposedApi` gate change is documented in xterm.js 5.4's release notes ("Remove check proposed api checks") and matches the observed symptom (no error, just dead URLs), so the speculative cost was acceptable: worst case is no change, not a regression.
