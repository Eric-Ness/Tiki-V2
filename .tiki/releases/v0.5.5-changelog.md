# v0.5.5 — xterm package namespace migration

Single-issue release that carries forward the dependency-namespace migration deferred from v0.5.4. No user-visible behavior change — the same xterm.js code, just imported from the package name the addons already use.

## Other Changes

- **Migrate `xterm@5.3.0` → `@xterm/xterm@5.5.0`** ([#161](https://github.com/Eric-Ness/Tiki-V2/pull/168)). 5.3.0 was the last release on the unscoped `xterm` name; everything since (5.4+, 5.5+) ships only under `@xterm/xterm`. Addons (`@xterm/addon-fit`, `-web-links`, `-search`) were already on the scoped namespace, so this collapses the dual-namespace structural-typing setup — TypeScript jump-to-definition now resolves to the right surface, and we're positioned to pick up bug fixes / accessibility work from 5.4+ in future bumps.

## Why this release was its own bundle

v0.5.5 was scoped to a single mechanical issue so the release itself could test the path that had blocked it: regenerating `pnpm-lock.yaml` on this local Windows machine. That root cause turned out to be a **Windows 11 kernel-level reparse-point trust mitigation**, not Defender as previously documented. Junctions created by Medium-IL processes are rejected by the kernel as "untrusted mount points" regardless of antivirus state. The fix is to run `pnpm install` from an elevated PowerShell so the junctions are created at High IL; everything else (build, dev server, tests) continues to run from a normal shell. CLAUDE.md's Known Environment Gotchas section and the corresponding memory note have been updated.
