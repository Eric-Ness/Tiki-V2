# v0.6.2

A capability release: GitHub API rate-limit visibility, hands-off `/tiki:release`, sidebar auto-refresh on pipeline events, phase-summary view, kanban within-column reordering, and multi-select batch operations on issues.

## Features

- **GitHub API rate-limit handling** ([#110](https://github.com/Eric-Ness/Tiki-V2/issues/110)) ŌĆö New footer indicator shows core-bucket `remaining/limit` with severity colors (healthy Ōēź50%, warn 10ŌĆō50%, critical <10%); click/hover tooltip surfaces all three GitHub API buckets (core, search, graphql) plus reset times. Backend adds rate-limit detection (primary + secondary) and synchronous exponential-backoff retry on secondary rate limits across all 7 read commands. Centralizes the previously-duplicated `gh` CLI error formatting into one helper. Caching + universal write-path backoff deferred to a follow-up.

- **/tiki:release runs hands-off + parallel-wave dispatch** ([#195](https://github.com/Eric-Ness/Tiki-V2/issues/195)) ŌĆö Removes all `AskUserQuestion` prompts from the release cascade success path: scope confirmation, ship gating, post-ship next-actions are gone. Pause-on-failure semantics preserved (audit FAIL, phase verification fail, build/test fail, push fail still pause). Adds an `<independence-analysis>` section that classifies issues by hard `depends on #N` deps + soft file-conflict heuristics, then partitions them into waves. Each wave dispatches issues as parallel sub-agents with `isolation: \"worktree\"`; deterministic per-wave merge in ascending issue-number order, abort-on-conflict, configurable parallelism cap (default 3, `--max-parallel N` override).

- **Auto-refresh sidebars after pipeline events** ([#196](https://github.com/Eric-Ness/Tiki-V2/issues/196)) ŌĆö When state.json transitions move work from `activeWork` to history (issue close or release ship), the desktop's Issues / Releases / PRs sidebar lists auto-refresh within ~1s. Adds `detectGithubRefreshTriggers(prev, next)` pure-function transition detector + 500ms per-surface trailing-edge debounce so a 5-issue release ship collapses to one fetch per surface. Releases store gains the `refetchCounter` + `triggerRefetch` pattern that issues/PRs already had.

- **Phase summary view in IssueDetail** ([#93](https://github.com/Eric-Ness/Tiki-V2/issues/93)) ŌĆö New `<PhaseSummaries>` component mounted between the Pipeline Timeline and the PlanEditor surfaces the per-phase summaries that have been quietly recorded in plan JSON files. Native `<details>` for expand/collapse ŌĆö no custom toggle state. Empty-state copy for plans without summaries yet. Refactors IssueDetail's existing inline TikiPlanŌåÆEditorPlan conversion into a shared `useMemo` so PlanEditor and PhaseSummaries split the cost.

- **Kanban: drag-to-reorder cards within columns + reset** ([#97](https://github.com/Eric-Ness/Tiki-V2/issues/97)) ŌĆö Cards can now be reordered within a column via drag (cross-column drags retain their existing transition behavior). Order persists per-project, per-column via the kanban store (persist version 2 ŌåÆ 3 with migration). Pure `applyColumnOrder` helper handles the merge of persisted order with API order ŌĆö new issues append at the bottom. Per-column reset arrow appears in the header when a custom order is active.

- **Batch operations on issues ŌĆö selection store + close/label toolbar** ([#96](https://github.com/Eric-Ness/Tiki-V2/issues/96)) ŌĆö Multi-select via checkboxes on KanbanCard and the sidebar's IssueCard (with `onPointerDown` propagation stop so checking doesn't start a drag). New transient per-project `useSelectionStore`. Selection count + Clear button in the kanban filter row. Floating bottom-center `<BulkActionToolbar>` appears when selection > 0 ŌĆö bulk Close + bulk Add Label (with filterable `<LabelPickerPopover>`); both actions fan out via Promise.all and report per-issue failures via the existing toast surface, then trigger an issues-list refresh. Shift+click range, bulk remove-label, bulk YOLO, and bulk assign-to-release deferred to [#203](https://github.com/Eric-Ness/Tiki-V2/issues/203).
