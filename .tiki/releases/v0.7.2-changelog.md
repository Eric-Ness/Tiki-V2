# v0.7.2 — Single `activeWork` store

Wave 3 (part 2, final) of the status-desync epic ([#218](https://github.com/Eric-Ness/Tiki-V2/issues/218)) — the high-risk refactor held back from v0.7.1, verified live before ship. Completes the epic: all three waves (Kanban Completed fix, single source of truth + GitHub freshness, write-integrity + store consolidation) are now in.

## Changed
- **Collapse the dual `activeWork` store onto `tikiStateStore`** ([#223](https://github.com/Eric-Ness/Tiki-V2/issues/223)) — There were two copies of `activeWork` kept in sync by hand: a React-local `App.state.activeWork` and `tikiStateStore.activeWork`, with the sidebar reading one and the kanban the other, so they could diverge. Every surface — sidebar (`StateSection`), kanban, detail (`selectedIssueWork`), and stale-work detection — now reads `tikiStateStore` as the single source. The store seeds `activeWork` to a stable `{}` ref, so the inline `?? {}` fallback (the fresh-object-per-render pattern behind the #210/#212 "Maximum update depth" render-loop crash class) is gone, and the dead `EMPTY_ACTIVE_WORK` constant was removed. The React-local copy is retained only for the load guard + change-detection diff, not as a render source. Because the failure mode is a runtime render loop that unit tests don't catch, this shipped only after a live `tauri:dev` verification gate on top of the static `pnpm build` + vitest gates.
