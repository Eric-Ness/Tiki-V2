# v0.7.0 — Status desync, Wave 2

Wave 2 of the [#218](https://github.com/Eric-Ness/Tiki-V2/issues/218) status-desync epic. Where v0.6.7 (Wave 1) fixed release-shipped issues never reaching the Kanban Completed column, this release attacks the **master cause**: ~4 UI surfaces each derived "status" independently from ~5 in-flight sources, so the same work item could render three different states at once. v0.7.0 introduces a single source of truth and a GitHub freshness mechanism so every surface agrees — and stays current.

## Features
- **Single source of truth — `deriveDisplayStatus()` selector** ([#222](https://github.com/Eric-Ness/Tiki-V2/issues/222)) — One pure selector implements the canonical first-match precedence table from #218 and returns `{ column, label, pipelineState, badge, anomaly? }`. The Kanban board, detail GitHub badge, and pipeline timeline now all consume it instead of re-deriving status, so they agree by construction. GitHub/Tiki disagreement (e.g. "shipping" in Tiki while GitHub is still open) is surfaced as an explicit anomaly rather than silently resolved. A new fresh-ref guard test protects against re-introducing the #210/#212 crash class.
- **GitHub freshness — re-sync on window focus** ([#221](https://github.com/Eric-Ness/Tiki-V2/issues/221)) — The watcher only watched `.tiki/`, so issues/PRs/releases closed outside the app (PR-merge auto-close, `gh` CLI, github.com) stayed stale until a manual refresh. The app now re-syncs on window focus / tab visibility (on by default) and offers an opt-in periodic poll (off by default, clamped to a 60s floor and gated by the GitHub rate-limit budget). Settings live in the GitHub section.

## Bug Fixes
- **Frozen display across surfaces** ([#220](https://github.com/Eric-Ness/Tiki-V2/issues/220)) — Three correctness fixes: the detail panel re-fetches the selected issue on every state change so a close-elsewhere updates the badge without reselection; the Rust file watcher switched from leading-edge to trailing-edge debounce so the *final* write of a burst (e.g. a release teardown) is always emitted instead of freezing the UI on an intermediate state; and stale `work.phase` ("1/N") is now cleared on completion in both the Node shim and the Rust IPC, guarded by a parity test.
