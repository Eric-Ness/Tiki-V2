# v0.5.0

Theme: **framework polish.** Close the remaining direct-JSON exceptions in the `state.mjs` shim, add deterministic AUDIT-time checks for coverage and dependency cycles, and enforce three-way parity on the canonical `WorkStatus` transition table so the TS / Rust / JS mirrors cannot silently drift.

This is a framework-internal release — the desktop app's visible surface is unchanged. The value is in shrinking the prose contracts and removing a class of latent bugs.

## Added

- **`state.mjs get <work-id> [--field <path>]`** (E27) — bash-friendly read path with dot-path field extraction. Scalars print raw (`executing\n`) and objects/arrays print as JSON, matching `jq -r` semantics. Replaces inline `cat | jq` invocations across framework commands.

- **`state.mjs remove <work-id>`** (E25) — atomic, validated removal of an `activeWork` entry. Closes the standalone-ship direct-JSON exception previously documented at `ship.md:228`.

- **`state.mjs append-history <issue|release> ...`** (E26) — typed appender for `history.recentIssues` / `history.recentReleases` plus the matching `lastCompletedIssue` / `lastCompletedRelease` fields. Shapes are taken straight from `completedIssueRecord` / `completedReleaseRecord` in `state.schema.json`. Closes the history-append direct-JSON exceptions in both `ship.md:231` and `release.md:168`.

- **`state.mjs --dry-run`** (E28) — apply-in-memory + stdout-only output, with no `state.json` write. Honored by `transition`, `remove`, and `append-history`. Illegal transitions still exit 1, so this is safe for preview before committing.

- **AUDIT-time algorithmic checks** (E31, E32) — `audit.md` now has a deterministic `<algorithmic-checks>` block that runs (a) bidirectional `successCriteria` ↔ `coverageMatrix` coverage matching, and (b) Kahn's algorithm on phase `dependencies` to surface cycles at plan-time rather than mid-execution. Both produce specific FAIL diagnostics with templated messages.

- **`packages/shared/src/__tests__/transitions-parity.test.ts`** (E44) — parses `state.mjs`'s `LEGAL` block and `state_transition.rs`'s `is_legal_transition` match arms, then compares pair-for-pair against the canonical `VALID_TRANSITIONS` table in `@tiki/shared`. If any of the three drift, this test fails with a specific status-pair that disagrees. Replaces the previous "must be kept in sync" comments with mechanical enforcement.

## Changed

- **`scripts/version-bump.mjs`** (E45) — now bumps the root `package.json` alongside `tauri.conf.json`, `Cargo.toml`, and `plugin.json`. Root had been stuck at `0.1.0` since project creation.

- **`ship.md` and `release.md` state-management sections** — rewritten to call `state.mjs remove` and `state.mjs append-history` instead of describing direct JSON writes. The prose "direct JSON; shim does not expose deletion yet" apology is gone.

- **`package.json` root `test:ts` script** — now runs `pnpm -C packages/shared test && pnpm -C apps/desktop test` instead of just the apps/desktop tests. The existing `transitions.test.ts` (and the new `transitions-parity.test.ts`) were silently unrun in CI before this — they did not actually enforce anything. CI now executes them.

- **`.github/workflows/pr.yml`** — adds a `Vitest (packages/shared)` step alongside the existing `Vitest (apps/desktop)` step so the shared-package tests run on every PR.

## Notes

- All four new `state.mjs` subcommands share the same `--tiki-path` override and the same exit-code contract (`0` success, `1` validation error, `2` I/O failure) as `transition`. Adding new subcommands later should follow the same pattern.

- The Kahn's check in `audit.md` is intentionally aligned with the algorithm in `execute.md`'s `<parallel-execution>` Step 2 — same semantics, same failure mode. The audit version surfaces cycles before any sub-agent dispatch happens.

- The three-way parity test reads source files directly (regex-based) rather than importing them as modules. That's intentional: `state.mjs` runs `main()` on import (a CLI), and `state_transition.rs` is Rust — there is no clean import path. Regex parsing is brittle if either file's formatting drastically changes, but the format is stable today and the test's failure message will point at the exact disagreement.
