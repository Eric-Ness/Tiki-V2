# v0.5.1

Theme: **dependency graph fixes + v2 visuals**. Closes a long-standing visibility gap in the desktop app's Dependency Graph view (the release dropdown was stuck at v0.2.15, the version sort was broken, edges were missing because the parser was too strict), then layers on the first round of visual upgrades from the v3 brainstorm: swimlanes, in-node progress fills, and hover-to-trace lineage.

This release is also the first to ship `/tiki:release`'s create-if-absent fix end-to-end, closing the workflow gap that left v0.4.1, v0.4.2, and v0.5.0 without release JSON files on disk.

## Fixed

- **Release JSON visibility gap end-to-end** (#147) — `/tiki:release` now auto-creates `.tiki/releases/{version}.json` from a minimal scaffold when the file is absent (previously it errored out with `release-not-found`; the JSON was only ever written by the desktop UI's `save_tiki_release` Tauri IPC, so any release shipped issue-by-issue silently skipped creating it). `load_tiki_releases` (commands.rs:216) gained an `include_archived` flag (default `false` — preserves #142's sidebar fix; the Dependency Graph passes `true`). Backfilled v0.4.1, v0.4.2, v0.5.0 JSONs in `archive/` (these were E-tracked enhancement bundles with no GitHub issue numbers — `issues: []` is the honest shape). Added a startup parity warn when changelog count exceeds JSON count under `.tiki/releases/`. Also committed the `<algorithmic-checks>` block in `audit.md` that was the actual v0.5.0 E31/E32 work sitting uncommitted in the working tree.

- **Dependency graph version dropdown sorts semver-correctly** (#148) — `DependencyGraph.tsx:30` was calling `b.version.localeCompare(a.version)`, which is a string compare: `"v0.2.9"` sorted as greater than `"v0.2.15"` because `'9' > '1'` lexicographically. The Rust backend already returns releases pre-sorted via `cmp_semver` (commands.rs:251) — the frontend's re-sort was undoing it. Dropped the `localeCompare`; rely on `Array.prototype.sort` stability to preserve the backend's order. Mirrors the resolution pattern from #120 (sidebar `ReleasesSection`); the dependency-graph component was a forgotten second consumer of the same bug.

- **Dep graph parser now finds markdown 'Related:' sections** (#150) — `parseDependencies.ts` previously only recognized `(?:depends on|blocked by|requires|after)\s+#(\d+)` directly inline. Real issue bodies in this project almost always use a markdown `## Related` section with a bulleted list, which the tight regex couldn't traverse. Result: most release graphs were disconnected dots and the new hover-lineage feature had nothing to trace. The parser now does three passes: inline tight patterns (existing 4 plus `related to` and `see also`), section-based scan that captures any `## related`-headed block up to the next heading and harvests every `#N` reference inside, and the existing `releaseIssueNumbers` filter. No edge-type distinction yet between hard deps and soft refs — both render as regular edges. That's left for a future v3 polish.

## Added

- **Dependency Graph v2 — three visual upgrade layers** (#149):
  - **Status swimlane backgrounds** (`SwimlaneLayer.tsx`, new). Mounted as a `<ReactFlow>` child. Groups laid-out nodes by their dagre rank, resolves the dominant status per rank, and paints a low-alpha colored band behind each. Tracks pan/zoom by reading viewport transform from xyflow's zustand store.
  - **In-node phase progress fills**. `IssueNodeData` gained an optional `phaseProgress: { current, total }`. `useDependencyGraph` surfaces it from `activeWork[issue:N].phase`. `IssueNode` renders a 3px progress bar at the bottom edge: full green for completed/closed, partial blue with a soft glow for executing, hidden otherwise. Doesn't change node dimensions.
  - **Hover-to-trace lineage**. On node hover, the hovered node plus its full ancestor and descendant chain stay at full opacity with a subtle accent ring; everything else dims to ~30%. Adjacency maps memoized from edges; BFS forward + backward yields the lineage set. Takes precedence over Critical Path mode when both are active; hover-off restores critical-path styling cleanly.

## Changed

- **`.claude/commands/tiki/` mirror sync** — earlier v0.5.0 prose work was uncommitted in the working tree at session start. The `release.md` and `ship.md` prose-to-shim conversions (replacing direct-JSON descriptions with concrete `state.mjs` subcommand calls) shipped as a separate `chore: finalize v0.5.0 framework prose in .claude/ mirror` commit. Resulting `.claude/` copies are byte-identical to `packages/framework/commands/`.

- **`apps/desktop/src-tauri/Cargo.lock`** — incidental: `tiki-desktop` package version drifted from `0.3.0` to `0.5.0` to match `Cargo.toml`. Picked up by cargo's auto-resolution during this session's `cargo check` runs. No functional impact.

- The Windows pnpm reparse-point block (CLAUDE.md) was still in effect during this session, so `pnpm build` could not be run end-to-end. Rust changes were verified via `cargo check` (3-7s incremental). TypeScript-side changes (the dep-graph component and SwimlaneLayer) were verified by inspection only — they're surface-localized and reversible per-phase if any visual issue surfaces.
- #147's hypothesis was originally framed as "regression in release.md/ship.md" but a REVIEW sub-agent's investigation overturned that: the JSON write step was never in those files; the gap was a workflow design hole. The reframe is captured in the updated issue body and the eventual fix structure.
