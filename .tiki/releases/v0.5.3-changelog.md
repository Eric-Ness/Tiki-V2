# v0.5.3

Theme: **terminal paste fix + CI hotfix**. A narrow patch release: one user-visible bug fix, one latent-since-February CI gate fix, and a quiet cure of the version-drift bug that had every binary since v0.5.0 reporting itself as `0.5.0` internally regardless of git tag.

This was originally planned as a 7-issue "Terminal polish" bundle. The local build verification path is currently blocked on this machine by Windows Defender filesystem-filter rejecting access through pnpm's `.pnpm/` junction symlinks (see CLAUDE.md / MEMORY.md `windows_pnpm_reparse_point`), so the six other issues (#156-#161) were split out to v0.5.4 and the paste fix was shipped solo. The clippy fix in `github.rs` rode along because every future PR would have tripped on it otherwise.

## Fixed

- **Ctrl+V no longer double-pastes** (#155) — The custom `attachCustomKeyEventHandler` branches in `apps/desktop/src/components/terminal/Terminal.tsx` for Ctrl+V and Ctrl+Shift+V called `xterm.paste(clipboardText)` and returned `false` "to prevent xterm from processing." But that return value only suppresses xterm's keyboard pipeline; the browser's separate `paste` DOM event still fires on xterm's hidden textarea, and xterm's built-in paste listener writes the same text a second time. Both custom branches deleted; xterm.js's native paste path now owns paste exclusively. The Ctrl+Shift+C copy branch stays — xterm doesn't provide it natively. The Wispr-compatibility comment was based on a misconception (Wispr writes to the OS clipboard, which xterm's native handler reads the same way).

- **`cargo clippy --deny warnings` no longer fails on Linux CI** — `hidden_command` in `apps/desktop/src-tauri/src/github.rs` declared `let mut cmd = Command::new(...)` but only mutated `cmd` inside the `#[cfg(target_os = "windows")]` block (for `creation_flags`). On Linux the `cfg` block is excluded, leaving `cmd` unmutated, which clippy with `-D warnings` rejects via `unused_mut`. Latent since #121 (Feb 2026); the lint surfaced for the first time today because the PR CI workflow from #145 had never actually fired on a PR before — the v0.5.3 paste-doubling PR was its first run. Scoped the allow to non-Windows targets so the lint stays enforced on Windows where the `mut` is genuinely used.

## Notes

- **Version drift cured.** `tauri.conf.json`, `Cargo.toml`, root `package.json`, and `plugin.json` had all been pinned at `0.5.0` since the v0.5.0 release — `pnpm version-bump` was apparently never run during the v0.5.1 or v0.5.2 ship steps, so every binary published in this version range reported itself as `0.5.0` to the auto-updater regardless of git tag. Bumped to `0.5.3` as part of this release. Worth auditing the `/tiki:release` flow to understand why the bump was missed for two consecutive releases.
- **PR CI gate was never live.** The "PR" workflow (`.github/workflows/pr.yml`) introduced in #145 on 2026-05-11 ran for the first time today against this PR. Issues #146 through #154 each had a PR merged to main without the workflow ever firing — either branch-protection didn't require it, or the merges bypassed the `pull_request` trigger somehow. Worth a separate follow-up; the workflow is a paper tiger right now even though it just happened to wake up and catch the latent clippy issue.
- **No build verification was possible locally.** `pnpm build` (which runs `tsc -b`, the strict project-references mode) couldn't run on this machine due to the Defender block. The paste fix is small enough to ship by-inspection. The clippy fix was reasoned through, not empirically verified locally — CI on the next PR will be the test of record.
