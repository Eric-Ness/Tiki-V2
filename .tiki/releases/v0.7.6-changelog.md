# v0.7.6 — reconciler reaches installed projects

Follow-up to v0.7.5. The enforced pipeline-tracking reconciler now works in **any installed project**, not just the Tiki repo itself — delivered via whichever channel installed the framework.

## Added
- **Two-channel reconciler distribution** ([#251](https://github.com/Eric-Ness/Tiki-V2/issues/251)):
  - **Plugin channel** — the Tiki plugin now ships `hooks/hooks.json` defining `Stop` + `SubagentStop` hooks that run the reconciler via `${CLAUDE_PLUGIN_ROOT}` (which expands reliably in hook commands, including on Windows). The scripts already ship with the plugin.
  - **Copy-install channel** — the desktop's framework install (and `install.js`) now install the framework scripts to `<project>/.claude/tiki/scripts/` and register the reconciler `Stop`/`SubagentStop` hook in the project's `.claude/settings.json`. The hook write is idempotent and never clobbers existing settings or other hooks.

## Changed
- **Plugin renamed `tiki-framework` → `tiki`** so commands namespace as `/tiki:*` when Tiki is loaded as a Claude Code plugin.
- **Command files now reference scripts at `.claude/tiki/scripts/`** (the installed location) instead of a monorepo-only path, with a direct-JSON fallback for plugin-only projects where that path isn't present.

## Internal
- New guard tests for the plugin manifest / hooks-file validity and the installed-script parity (a stand-in for `claude plugin validate`), plus Rust unit tests for the idempotent hook-merge.

## Note
- The reconciler hooks (plugin-defined and desktop-written) require approval in Claude Code before they run — by design, since hooks execute code.
